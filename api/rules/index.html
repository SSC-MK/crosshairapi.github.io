    Rules Engine¶

    The Rules API provides a declarative rule engine with three subsystems: Declarative Rulesets for automated content extraction, Sigma Rules for log analysis and SIEM integration, and YARA Rules for binary pattern matching.

    Declarative Rulesets¶

    Manage declarative rulesets for automated content extraction and actions. Rulesets target specific domains and contain rule groups with URL patterns and scraping/action rules.

    List Rulesets¶

    GET /v1/rules

    List all loaded rulesets, optionally filtered by domain.

    Parameters:

    Name Type Default Description
    domain string -- Filter by target domain
    enabled_only boolean true Only return enabled rulesets 
    curl "http://localhost:8001/v1/rules?domain=example.com"

    curl "http://localhost:8001/v1/rules?enabled_only=false"
    Response 
    {
  "success": true,
  "data": {
    "rulesets": [
      {
        "id": "rs_abc123",
        "name": "Example Ruleset",
        "description": "Extract data from example.com",
        "target_domains": ["example.com"],
        "rule_groups_count": 3,
        "created_at": "2026-01-15T10:30:00Z",
        "updated_at": "2026-01-20T14:00:00Z"
      }
    ],
    "total": 1
  }
}

    Create Ruleset¶

    POST /v1/rules

    Create a new ruleset. You can provide either yaml_content (raw YAML string) or structured rule_groups.

    curl -X POST http://localhost:8001/v1/rules \
  -H "Content-Type: application/json" \
  -d '{
    "ruleset_name": "My Ruleset",
    "description": "Extract product data",
    "target_domains": ["shop.example.com"],
    "yaml_content": "ruleset_name: My Ruleset\nrule_groups:\n  - group_name: products\n    url_patterns:\n      - \"/product/*\"\n    scraping_rules:\n      - field: title\n        selector: h1.product-title"
  }'
    Response 
    {
  "success": true,
  "data": {
    "id": "rs_def456",
    "name": "My Ruleset",
    "rule_groups_count": 1,
    "message": "Ruleset created successfully"
  }
}

    Get Ruleset¶

    GET /v1/rules/{ruleset_id}

    Retrieve a ruleset by its ID.

    Parameters:

    Name Type Default Description
    include_yaml boolean false Include YAML representation in response 
    curl "http://localhost:8001/v1/rules/rs_abc123?include_yaml=true"

    Update Ruleset¶

    PUT /v1/rules/{ruleset_id}

    Update a ruleset. Only the provided fields are modified.

    Request Body:

    {
  "description": "Updated description",
  "target_domains": ["shop.example.com", "store.example.com"]
}

    curl -X PUT http://localhost:8001/v1/rules/rs_abc123 \
  -H "Content-Type: application/json" \
  -d '{"description": "Updated description", "target_domains": ["shop.example.com", "store.example.com"]}'

    Delete Ruleset¶

    DELETE /v1/rules/{ruleset_id}

    Delete a ruleset permanently.

    curl -X DELETE "http://localhost:8001/v1/rules/rs_abc123"

    Validate Ruleset¶

    POST /v1/rules/validate

    Validate YAML content without loading it into the rule engine. Useful for CI/CD pipelines or editor integrations.

    curl -X POST http://localhost:8001/v1/rules/validate \
  -H "Content-Type: application/json" \
  -d '{"yaml_content": "ruleset_name: Test\nrule_groups:\n  - group_name: test"}'
    Response 
    {
  "success": true,
  "data": {
    "valid": true,
    "errors": [],
    "warnings": ["No scraping_rules defined in group 'test'"]
  }
}

    Validation in CI/CD

    Use the validate endpoint to check rulesets before deploying them. A response with valid: true but non-empty warnings means the ruleset will load but may not behave as expected.

    Test Ruleset¶

    POST /v1/rules/test

    Test a ruleset against a live URL to preview extraction results.

    curl -X POST http://localhost:8001/v1/rules/test \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://example.com/product/123",
    "ruleset_id": "rs_abc123",
    "extract_data": true
  }'

    curl -X POST http://localhost:8001/v1/rules/test \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://example.com/product/123",
    "yaml_content": "ruleset_name: Test\nrule_groups:\n  - group_name: products\n    url_patterns:\n      - \"/product/*\"\n    scraping_rules:\n      - field: title\n        selector: h1",
    "extract_data": true
  }'
    Response 
    {
  "success": true,
  "data": {
    "url": "https://example.com/product/123",
    "ruleset_name": "Product Extractor",
    "matching_groups": 1,
    "matched_group_names": ["product_page"],
    "extraction_results": ["..."],
    "total_fields_extracted": 5
  }
}

    Match URL¶

    POST /v1/rules/match

    Find all rulesets that match a given URL. Useful for debugging which rules would apply to a page.

    Parameters:

    Name Type Description
    url string URL to match against all rulesets (required) 
    curl -X POST "http://localhost:8001/v1/rules/match?url=https://example.com/page"
    Response 
    {
  "success": true,
  "data": {
    "url": "https://example.com/page",
    "matches": [
      {
        "ruleset_id": "rs_abc123",
        "ruleset_name": "Example Ruleset",
        "matching_groups": [
          {
            "group_name": "main_content",
            "scraping_rules": 5,
            "action_rules": 2
          }
        ]
      }
    ],
    "total_rulesets": 1,
    "total_groups": 1
  }
}

    Sigma Rules¶

    Sigma is a generic and open signature format for SIEM systems. The Sigma Rules API allows you to validate, convert, parse, and match Sigma rules against log data.

    Supported Backends

    Sigma rules can be converted to queries for: Splunk, Elastic, QRadar, and native Sigma format.

    List Backends¶

    GET /v1/rules/sigma/backends

    List available Sigma conversion backends.

    curl http://localhost:8001/v1/rules/sigma/backends
    Response 
    {
  "success": true,
  "data": {
    "backends": ["splunk", "elastic", "qradar", "sigma"]
  }
}

    Validate Sigma Rule¶

    POST /v1/rules/sigma/validate

    Validate the syntax and structure of a Sigma rule.

    Request Body:

    {
  "rule": "title: Test Rule\nstatus: test\nlogsource:\n  product: windows\ndetection:\n  selection:\n    EventID: 4624\n  condition: selection"
}

    curl -X POST http://localhost:8001/v1/rules/sigma/validate \
  -H "Content-Type: application/json" \
  -d '{"rule": "title: Test Rule\nstatus: test\nlogsource:\n  product: windows\ndetection:\n  selection:\n    EventID: 4624\n  condition: selection"}'
    Response 
    {
  "success": true,
  "data": {
    "valid": true,
    "errors": [],
    "warnings": []
  }
}

    Convert Sigma Rule¶

    POST /v1/rules/sigma/convert

    Convert a Sigma rule to a backend-specific query language (e.g., Splunk SPL, Elastic Query DSL).

    Request Body:

    Field Type Description
    rule string Sigma rule in YAML format (required)
    backend string Target backend: splunk, elastic, qradar, sigma (required) 
    curl -X POST http://localhost:8001/v1/rules/sigma/convert \
  -H "Content-Type: application/json" \
  -d '{
    "rule": "title: Login Event\nstatus: test\nlogsource:\n  product: windows\ndetection:\n  selection:\n    EventID: 4624\n  condition: selection",
    "backend": "splunk"
  }'
    Response 
    {
  "success": true,
  "data": {
    "backend": "splunk",
    "query": "EventID=4624"
  }
}

    Parse Sigma Rule¶

    POST /v1/rules/sigma/parse

    Parse a Sigma rule and extract its metadata, detection logic, and structure.

    Request Body:

    {
  "rule": "title: Test Rule\nstatus: test\n..."
}

    curl -X POST http://localhost:8001/v1/rules/sigma/parse \
  -H "Content-Type: application/json" \
  -d '{"rule": "title: Test Rule\nstatus: test\nlogsource:\n  product: windows\ndetection:\n  selection:\n    EventID: 4624\n  condition: selection"}'

    Match Logs Against Sigma Rule¶

    POST /v1/rules/sigma/match

    Evaluate log events against a Sigma rule and return matching entries.

    Request Body:

    Field Type Description
    rule string Sigma rule in YAML format (required)
    logs array Array of log event objects to evaluate (required) 
    curl -X POST http://localhost:8001/v1/rules/sigma/match \
  -H "Content-Type: application/json" \
  -d '{
    "rule": "title: Login Event\nstatus: test\nlogsource:\n  product: windows\ndetection:\n  selection:\n    EventID: 4624\n  condition: selection",
    "logs": [
      {"EventID": 4624, "User": "admin"},
      {"EventID": 4625, "User": "guest"}
    ]
  }'
    Response 
    {
  "success": true,
  "data": {
    "matches": [
      {"index": 0, "log": {"EventID": 4624, "User": "admin"}}
    ],
    "total_logs": 2,
    "matched": 1
  }
}

    YARA Rules¶

    YARA is a tool for identifying and classifying malware samples via pattern matching. The YARA Rules API enables validation, compilation, and scanning of data or URLs against YARA rules.

    Engine Info¶

    GET /v1/rules/yara/info

    Check YARA engine availability and version.

    curl http://localhost:8001/v1/rules/yara/info
    Response 
    {
  "success": true,
  "data": {
    "available": true,
    "version": "4.3.0"
  }
}

    Validate YARA Rule¶

    POST /v1/rules/yara/validate

    Validate a YARA rule for syntax correctness.

    Request Body:

    {
  "rule": "rule test_rule { strings: $a = \"test\" condition: $a }"
}

    curl -X POST http://localhost:8001/v1/rules/yara/validate \
  -H "Content-Type: application/json" \
  -d '{"rule": "rule test_rule { strings: $a = \"test\" condition: $a }"}'

    Compile YARA Rule¶

    POST /v1/rules/yara/compile

    Compile a YARA rule and return compilation statistics. Useful for checking rule complexity.

    Request Body:

    {
  "rule": "rule test_rule { strings: $a = \"test\" condition: $a }"
}

    curl -X POST http://localhost:8001/v1/rules/yara/compile \
  -H "Content-Type: application/json" \
  -d '{"rule": "rule test_rule { strings: $a = \"test\" condition: $a }"}'

    Scan Data¶

    POST /v1/rules/yara/scan

    Scan raw data (base64-encoded) against a YARA rule.

    Request Body:

    Field Type Description
    rule string YARA rule source (required)
    data string Data to scan (required)
    encoding string Encoding of data field: base64, utf-8 
    curl -X POST http://localhost:8001/v1/rules/yara/scan \
  -H "Content-Type: application/json" \
  -d '{
    "rule": "rule test_rule { strings: $a = \"malware\" condition: $a }",
    "data": "base64-encoded-data",
    "encoding": "base64"
  }'
    Response 
    {
  "success": true,
  "data": {
    "matches": [
      {
        "rule": "test_rule",
        "strings": [
          {"identifier": "$a", "offset": 10, "data": "malware"}
        ],
        "meta": {}
      }
    ],
    "scanned_bytes": 1024,
    "matched": true
  }
}

    Scan URL Content¶

    POST /v1/rules/yara/scan/url

    Fetch content from a URL and scan it against a YARA rule.

    Request Body:

    Field Type Description
    rule string YARA rule source (required)
    url string URL whose content will be scanned (required) 
    curl -X POST http://localhost:8001/v1/rules/yara/scan/url \
  -H "Content-Type: application/json" \
  -d '{
    "rule": "rule detect_php { strings: $a = \"<?php\" condition: $a }",
    "url": "https://example.com/script.php"
  }'
