    Network Endpoints¶

    The Network API provides a comprehensive suite for API traffic capture, reverse engineering, documentation generation, and monitoring. Using browser automation and proxy interception, it can discover hidden endpoints, generate client code in multiple languages, export to industry-standard formats, and continuously monitor APIs for breaking changes.

    Capabilities Overview¶

    Capability Description
    Traffic Capture Intercept browser network requests via Playwright
    API Discovery Smart endpoint detection with authentication inference
    Export Formats OpenAPI 3.0, Postman v2.1, Insomnia v4, HAR 1.2
    Command Generation curl, HTTPie, Python, JavaScript, PowerShell
    GraphQL Introspection Schema discovery for GraphQL APIs
    API Monitoring Track changes, detect breaking changes, webhook alerts
    Proxy Mode Capture traffic from any application (mobile, desktop, CLI)
    Security Testing Fuzzing, auth bypass testing, request replay

    POST /v1/network/capture¶

    Capture API traffic from a web page using browser automation. Opens the target URL in a headless browser, intercepts all network requests, and returns the discovered API endpoints.

    Best For

    Reverse engineering web APIs, discovering hidden endpoints, mapping attack surfaces.

    Target Audience

    • Security researchers -- Mapping attack surfaces
    • API developers -- Understanding third-party integrations
    • QA engineers -- Documenting API behavior

    Parameters¶

    Name Type Default Description
    url string -- URL to capture traffic from (required)
    wait_time float 3.0 Seconds to wait for network activity
    scroll boolean false Scroll page to trigger lazy loading
    click_selectors array -- CSS selectors to click (triggers additional requests)
    filter_hosts array -- Only capture traffic to these hosts
    exclude_hosts array -- Exclude traffic to these hosts

    Examples¶

    curl -X POST http://localhost:8001/v1/network/capture \
  -H "Content-Type: application/json" \
  -d '{"url": "https://jsonplaceholder.typicode.com", "wait_time": 5}'

    curl -X POST http://localhost:8001/v1/network/capture \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://example.com/app",
    "wait_time": 5.0,
    "scroll": true,
    "click_selectors": [".load-more"],
    "filter_hosts": ["api.example.com"]
  }'

    Response¶

    {
  "success": true,
  "data": {
    "url": "https://jsonplaceholder.typicode.com",
    "total_requests": 3,
    "api_endpoints": [
      {
        "url": "https://jsonplaceholder.typicode.com/posts",
        "method": "GET",
        "path": "/posts",
        "response": {"status": 200, "content_type": "application/json"}
      }
    ],
    "unique_domains": ["jsonplaceholder.typicode.com"],
    "auth_headers_found": {}
  }
}

    Response Fields¶

    Field Type Description
    url string The captured URL
    total_requests integer Total network requests intercepted
    api_endpoints array Discovered API endpoints with method, path, and response info
    unique_domains array All unique domains observed in traffic
    auth_headers_found object Authentication headers detected (e.g., Bearer tokens, API keys)

    POST /v1/network/discover¶

    Smart API endpoint discovery with authentication detection. Goes beyond simple traffic capture by analyzing patterns and detecting auth mechanisms.

    Parameters¶

    Name Type Default Description
    url string -- URL to discover APIs from (required)
    include_subdomains boolean false Include subdomain traffic
    detect_auth boolean false Detect authentication methods

    Example¶

    curl -X POST http://localhost:8001/v1/network/discover \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://api.example.com",
    "include_subdomains": true,
    "detect_auth": true
  }'

    POST /v1/network/generate/commands¶

    Generate copy-paste ready API commands in multiple languages and formats. Takes captured endpoint data and produces ready-to-use commands for testing, documentation, or automation.

    Best For

    Quick testing, sharing API examples, creating documentation snippets.

    Target Audience

    • Developers -- Testing endpoints quickly
    • Technical writers -- Creating API documentation
    • DevOps engineers -- Automating API calls

    Parameters¶

    Name Type Default Description
    endpoints array -- Endpoints to generate commands for (required)
    formats array -- Output formats: curl, httpie, python, javascript, powershell
    include_headers boolean true Include headers in output

    Examples¶

    curl -X POST http://localhost:8001/v1/network/generate/commands \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {
        "url": "https://api.github.com/users/octocat",
        "method": "GET"
      }
    ],
    "formats": ["curl"]
  }'

    curl -X POST http://localhost:8001/v1/network/generate/commands \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {
        "url": "https://api.example.com/users",
        "method": "POST",
        "request": {
          "headers": {
            "Authorization": "Bearer token",
            "Content-Type": "application/json"
          },
          "body": "{\"name\": \"John\"}"
        }
      }
    ],
    "formats": ["curl", "httpie", "python", "javascript", "powershell"]
  }'

    Response¶

    {
  "success": true,
  "data": {
    "results": [
      {
        "endpoint": "GET /users/octocat",
        "commands": {
          "curl": "curl https://api.github.com/users/octocat"
        }
      }
    ],
    "batch_script": "#!/bin/bash\ncurl https://api.github.com/users/octocat"
  }
}

    Below are examples of the output produced for a POST /users endpoint with authentication:

    Export Endpoints¶

    Export captured API endpoints to industry-standard formats for use in API testing tools, documentation generators, and CI/CD pipelines.

    POST /v1/network/export/openapi¶

    Generate an OpenAPI 3.0 specification from captured endpoints.

    Best For

    API documentation, SDK generation, contract testing.

    Parameters¶

    Name Type Required Description
    endpoints array Yes Captured endpoints to export
    title string No API title
    version string No API version string
    description string No API description

    Example¶

    curl -X POST http://localhost:8001/v1/network/export/openapi \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {
        "url": "https://api.example.com/users",
        "method": "GET",
        "response": {"status": 200, "content_type": "application/json"}
      }
    ],
    "title": "My API",
    "version": "1.0.0",
    "description": "API documentation"
  }'

    Response¶

    {
  "success": true,
  "data": {
    "openapi": "3.0.3",
    "info": {"title": "My API", "version": "1.0.0"},
    "paths": {
      "/users": {
        "get": {"responses": {"200": {"description": "OK"}}}
      }
    }
  }
}

    POST /v1/network/export/postman¶

    Export endpoints as a Postman Collection v2.1 file. The output is directly importable in Postman.

    Best For

    API testing in Postman, team collaboration on API collections.

    Parameters¶

    Name Type Required Description
    endpoints array Yes Captured endpoints to export
    name string No Collection name

    Example¶

    curl -X POST http://localhost:8001/v1/network/export/postman \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {
        "url": "https://api.example.com/users",
        "method": "GET"
      }
    ],
    "name": "My API Collection"
  }'

    POST /v1/network/export/insomnia¶

    Export endpoints as Insomnia Export v4 format, including optional environment variables.

    Best For

    API testing in Insomnia, organized request management.

    Parameters¶

    Name Type Required Description
    endpoints array Yes Captured endpoints to export
    name string No Workspace name
    include_environment boolean No Include environment variables

    Example¶

    curl -X POST http://localhost:8001/v1/network/export/insomnia \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {
        "url": "https://api.example.com/users",
        "method": "GET"
      }
    ],
    "name": "My API",
    "include_environment": true
  }'

    Response¶

    {
  "success": true,
  "data": {
    "_type": "export",
    "__export_format": 4,
    "resources": [
      {"_id": "wrk_xxx", "_type": "workspace", "name": "My API"},
      {"_id": "env_xxx", "_type": "environment", "name": "Base Environment"},
      {"_id": "req_xxx", "_type": "request", "name": "GET /users", "url": "..."}
    ]
  }
}

    POST /v1/network/export/har¶

    Export endpoints as HAR (HTTP Archive) 1.2 format for browser DevTools import and traffic analysis.

    Best For

    Browser DevTools import, traffic analysis, performance profiling.

    Parameters¶

    Name Type Required Description
    page_url string No Page URL context for the HAR file
    endpoints array Yes Captured endpoints to export

    Example¶

    curl -X POST http://localhost:8001/v1/network/export/har \
  -H "Content-Type: application/json" \
  -d '{
    "page_url": "https://example.com",
    "endpoints": [
      {
        "url": "https://api.example.com/users",
        "method": "GET",
        "response": {"status": 200}
      }
    ]
  }'

    Export Format Comparison¶

    Format Version Use Case Import Tool
    OpenAPI 3.0.3 API documentation, SDK generation, contract testing Swagger UI, Redoc, Stoplight
    Postman v2.1 Manual API testing, team collaboration Postman App
    Insomnia v4 Request management, environment variables Insomnia App
    HAR 1.2 Traffic replay, performance analysis Browser DevTools, Charles Proxy

    POST /v1/network/graphql/introspect¶

    Introspect a GraphQL API to discover its full schema, including types, queries, mutations, and subscriptions.

    Best For

    GraphQL API exploration, schema documentation, client code generation.

    Parameters¶

    Name Type Required Description
    endpoint_url string Yes GraphQL endpoint URL
    headers object No Custom headers (e.g., authentication)

    Example¶

    curl -X POST http://localhost:8001/v1/network/graphql/introspect \
  -H "Content-Type: application/json" \
  -d '{"endpoint_url": "https://countries.trevorblades.com/graphql"}'

    Response¶

    {
  "success": true,
  "data": {
    "introspection_enabled": true,
    "schema": {
      "queryType": {"name": "Query"},
      "types": [
        {"kind": "OBJECT", "name": "Country", "fields": ["..."]},
        {"kind": "OBJECT", "name": "Continent", "fields": ["..."]}
      ]
    }
  }
}

    Response Fields¶

    Field Type Description
    introspection_enabled boolean Whether the GraphQL endpoint allows introspection
    schema object Full GraphQL schema including types, queries, and mutations
    schema.queryType object Root query type
    schema.types array All types defined in the schema

    POST /v1/network/detect/auth¶

    Detect authentication methods used by captured API endpoints. Analyzes headers, cookies, and query parameters to identify auth patterns.

    Best For

    Security auditing, API documentation, understanding third-party authentication flows.

    Parameters¶

    Name Type Required Description
    endpoints array Yes Captured endpoints with request headers

    Example¶

    curl -X POST http://localhost:8001/v1/network/detect/auth \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {"request": {"headers": {"Authorization": "Bearer eyJ..."}}}
    ]
  }'

    Response¶

    {
  "success": true,
  "data": {
    "authentication_methods": [
      {"type": "bearer", "location": "header", "name": "Authorization", "confidence": 1.0},
      {"type": "api_key", "location": "header", "name": "X-API-Key", "confidence": 0.9}
    ]
  }
}

    Response Fields¶

    Field Type Description
    authentication_methods array Detected auth methods
    authentication_methods[].type string Auth type: bearer, api_key, basic, oauth2, cookie
    authentication_methods[].location string Where the auth is sent: header, query, cookie
    authentication_methods[].name string Header/parameter name
    authentication_methods[].confidence float Detection confidence (0.0 - 1.0)

    POST /v1/network/infer/schema¶

    Infer a JSON schema from API response samples. Provide multiple response examples and the endpoint will generate a unified schema.

    Parameters¶

    Name Type Required Description
    samples array Yes Array of JSON response samples

    Example¶

    curl -X POST http://localhost:8001/v1/network/infer/schema \
  -H "Content-Type: application/json" \
  -d '{
    "samples": [
      {"id": 1, "name": "John", "email": "john@example.com"},
      {"id": 2, "name": "Jane", "email": "jane@example.com"}
    ]
  }'

    Monitoring Endpoints¶

    Track API changes over time, detect breaking changes, and receive webhook alerts when APIs are modified.

    POST /v1/network/monitor/create¶

    Create a new API monitor to track changes over time.

    Best For

    Detecting breaking API changes, compliance monitoring, SLA tracking.

    Parameters¶

    Name Type Required Description
    name string Yes Monitor name
    url string Yes API URL to monitor
    interval_minutes integer No Check interval in minutes
    alert_webhook string No Webhook URL for change alerts (e.g., Slack)

    Example¶

    curl -X POST http://localhost:8001/v1/network/monitor/create \
  -H "Content-Type: application/json" \
  -d '{
    "name": "GitHub API Monitor",
    "url": "https://api.github.com",
    "interval_minutes": 60,
    "alert_webhook": "https://hooks.slack.com/..."
  }'

    Response¶

    {
  "success": true,
  "data": {
    "api_id": "abc123",
    "name": "GitHub API Monitor",
    "url": "https://api.github.com",
    "interval_minutes": 60,
    "enabled": true,
    "next_check": "2026-02-04T09:00:00Z"
  }
}

    Response Fields¶

    Field Type Description
    api_id string Unique monitor identifier
    name string Monitor name
    url string Monitored URL
    interval_minutes integer Check interval
    enabled boolean Whether the monitor is active
    next_check string ISO 8601 timestamp of next scheduled check

    GET /v1/network/monitor/list¶

    List all API monitors.

    Example¶

    curl http://localhost:8001/v1/network/monitor/list

    GET /v1/network/monitor/{api_id}¶

    Get details of a specific monitor including snapshots and detected changes.

    Example¶

    curl http://localhost:8001/v1/network/monitor/abc123

    POST /v1/network/monitor/{api_id}/capture¶

    Manually trigger a snapshot capture for a monitor (outside its normal schedule).

    Example¶

    curl -X POST http://localhost:8001/v1/network/monitor/abc123/capture

    DELETE /v1/network/monitor/{api_id}¶

    Delete an API monitor and all its captured snapshots.

    Example¶

    curl -X DELETE http://localhost:8001/v1/network/monitor/abc123

    POST /v1/network/monitor/compare¶

    Compare two OpenAPI specifications and detect changes, including breaking changes. Ideal for CI/CD pipelines and API versioning workflows.

    Best For

    CI/CD pipelines, API versioning, breaking change detection.

    Parameters¶

    Name Type Required Description
    old_spec object Yes Previous OpenAPI specification
    new_spec object Yes New OpenAPI specification

    Example¶

    curl -X POST http://localhost:8001/v1/network/monitor/compare \
  -H "Content-Type: application/json" \
  -d '{
    "old_spec": {
      "paths": {
        "/users": {"get": {}},
        "/orders": {"delete": {}}
      }
    },
    "new_spec": {
      "paths": {
        "/users": {"get": {}, "post": {}}
      }
    }
  }'

    Response¶

    {
  "success": true,
  "data": {
    "total_changes": 2,
    "breaking_changes": 1,
    "is_breaking": true,
    "changes": [
      {
        "change_type": "endpoint_added",
        "severity": "info",
        "description": "New method: POST /users"
      },
      {
        "change_type": "endpoint_removed",
        "severity": "critical",
        "description": "Removed endpoint: DELETE /orders"
      }
    ]
  }
}

    Change Types¶

    Type Severity Description
    endpoint_added info New endpoint discovered
    endpoint_removed critical Endpoint removed (breaking change)
    response_schema_changed warning Response structure changed
    request_schema_changed warning Request parameters changed
    status_code_changed warning HTTP status code changed

    Breaking Changes

    Changes with critical severity indicate breaking changes that may affect API consumers. The is_breaking field in the response is true when any critical change is detected.

    Proxy Endpoints¶

    Start an HTTP proxy server to capture traffic from any application -- mobile apps, desktop apps, CLI tools, or any HTTP client. Clients must explicitly configure proxy settings (non-invasive).

    POST /v1/network/proxy/start¶

    Start an HTTP proxy server for traffic capture.

    Best For

    Capturing traffic from mobile apps, desktop apps, CLI tools -- any application that supports proxy configuration.

    Non-Invasive

    Clients must explicitly configure their proxy settings. The proxy does not intercept traffic automatically.

    Parameters¶

    Name Type Default Description
    port integer 8888 Proxy server port
    filter_hosts array -- Only capture traffic to these hosts
    exclude_hosts array -- Exclude traffic to these hosts

    Example¶

    curl -X POST http://localhost:8001/v1/network/proxy/start \
  -H "Content-Type: application/json" \
  -d '{
    "port": 8888,
    "filter_hosts": ["api.example.com"],
    "exclude_hosts": ["analytics.google.com"]
  }'

    Response¶

    {
  "success": true,
  "data": {
    "session_id": "abc123",
    "port": 8888,
    "status": "running"
  }
}

    POST /v1/network/proxy/stop¶

    Stop the active proxy session and retrieve captured traffic.

    Example¶

    curl -X POST http://localhost:8001/v1/network/proxy/stop

    GET /v1/network/proxy/status¶

    Get current proxy session status.

    Example¶

    curl http://localhost:8001/v1/network/proxy/status

    GET /v1/network/proxy/traffic¶

    Get captured traffic from active or stopped proxy session.

    Parameters¶

    Name Type Default Description
    session_id string -- Session ID (uses active session if omitted)
    as_endpoints boolean false Convert traffic to standard endpoint format

    Example¶

    curl "http://localhost:8001/v1/network/proxy/traffic?as_endpoints=true"

    GET /v1/network/proxy/instructions¶

    Get setup instructions for configuring the proxy on different platforms and tools.

    Example¶

    curl http://localhost:8001/v1/network/proxy/instructions

    Response¶

    {
  "success": true,
  "data": {
    "proxy_url": "http://192.168.1.100:8888",
    "macos": "export HTTP_PROXY=http://192.168.1.100:8888",
    "windows": "$env:HTTP_PROXY = \"http://192.168.1.100:8888\"",
    "linux": "export HTTP_PROXY=http://192.168.1.100:8888",
    "ios": "Settings > Wi-Fi > Configure Proxy > Manual",

    Security Testing Endpoints¶

    POST /v1/network/fuzz¶

    Security fuzzing for API endpoints. Sends various attack payloads to test for common vulnerabilities.

    Use Responsibly

    Only use fuzzing against APIs you own or have explicit permission to test.

    Parameters¶

    Name Type Required Description
    url string Yes Target URL to fuzz
    params object No Base query parameters
    payloads array No Payload types to use
    max_requests integer No Maximum number of fuzz requests

    Payload Types¶

    Payload Description Example
    sql_injection SQL injection payloads ' OR '1'='1, 1; DROP TABLE
    xss Cross-site scripting <script>alert(1)</script>
    command_injection OS command injection ; cat /etc/passwd
    path_traversal Directory traversal ../../../etc/passwd

    Example¶

    curl -X POST http://localhost:8001/v1/network/fuzz \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://httpbin.org/get",
    "params": {"search": "test"},
    "payloads": ["sql_injection", "xss"],
    "max_requests": 10
  }'

    POST /v1/network/test/auth-bypass¶

    Test for authentication bypass vulnerabilities by attempting requests without credentials, with modified tokens, and other bypass techniques.

    Example¶

    curl -X POST http://localhost:8001/v1/network/test/auth-bypass \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {
        "url": "https://api.example.com/admin/users",
        "method": "GET",
        "request": {"headers": {"Authorization": "Bearer eyJ..."}}
      }
    ]
  }'

    Replay Endpoints¶

    POST /v1/network/replay¶

    Replay a captured API request, optionally modifying headers, parameters, or body.

    Parameters¶

    Name Type Required Description
    endpoint object Yes Captured endpoint object to replay
    modify_headers object No Headers to add or override
    modify_params object No Query parameters to add or override

    Example¶

    curl -X POST http://localhost:8001/v1/network/replay \
  -H "Content-Type: application/json" \
  -d '{
    "endpoint": {
      "url": "https://api.example.com/users",
      "method": "GET",
      "request": {"headers": {"Authorization": "Bearer old_token"}}
    },
    "modify_headers": {"Authorization": "Bearer new_token"},
    "modify_params": {"page": "2"}
  }'

    POST /v1/network/replay/batch¶

    Replay multiple captured requests in sequence or parallel.

    Example¶

    curl -X POST http://localhost:8001/v1/network/replay/batch \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {"url": "https://api.example.com/users", "method": "GET"},
      {"url": "https://api.example.com/posts", "method": "GET"}
    ]
  }'

    Analysis Endpoints¶

    POST /v1/network/analyze¶

    Analyze captured endpoints for patterns, authentication mechanisms, API versioning, and common conventions.

    Example¶

    curl -X POST http://localhost:8001/v1/network/analyze \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {"url": "https://api.example.com/v2/users", "method": "GET"},
      {"url": "https://api.example.com/v2/posts", "method": "GET"}
    ]
  }'

    POST /v1/network/diff¶

    Compare two sets of API captures to find differences in endpoints, parameters, response schemas, or authentication.

    Example¶

    curl -X POST http://localhost:8001/v1/network/diff \
  -H "Content-Type: application/json" \
  -d '{
    "before": [
      {"url": "https://api.example.com/users", "method": "GET"}
    ],
    "after": [
      {"url": "https://api.example.com/users", "method": "GET"},
      {"url": "https://api.example.com/users", "method": "POST"}
    ]
  }'

    POST /v1/network/mock/generate¶

    Generate a mock server configuration from captured endpoints. Useful for frontend development, testing, and CI/CD environments.

    Example¶

    curl -X POST http://localhost:8001/v1/network/mock/generate \
  -H "Content-Type: application/json" \
  -d '{
    "endpoints": [
      {
        "url": "https://api.example.com/users",
        "method": "GET",
        "response": {"status": 200, "body": [{"id": 1, "name": "John"}]}
      }
    ]
  }'

    POST /v1/network/websocket/capture¶

    Capture WebSocket connections and messages from a webpage. Opens the target URL in a headless browser and records all WebSocket activity.

    Parameters¶

    Name Type Default Description
    url string -- URL to capture WebSocket traffic from (required)
    wait_time integer 10 Seconds to listen for WebSocket messages

    Example¶

    curl -X POST http://localhost:8001/v1/network/websocket/capture \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://example.com/app",
    "wait_time": 10
  }'
